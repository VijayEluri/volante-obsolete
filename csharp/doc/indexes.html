Volante : Volante Developer's Guide : Indexes

Storing objects is just half of the job. The other half is (quickly) retrieving objects based on some criteria. That's the job of indexes.

Similar to relational (SQL) databases, indexes need to be explicitly created. Unlike SQL, you have to add objects to indexes explicitly.

A given object can belong to multiple indexes.

Indexes only store object's oid (32-bit integer) to refer to the object, so no duplicate data is created when an object is inserted into multiple indexes.

Indexes are usually part of the root object.

Indexes are persistent objects (objects implementing IPersistent interface) themselves, so they can be used in all context where IPersistent class can be used.

Inserting an object into index makes it reachable (assuming the index itself is reachable from other reachable object e.g. root object) which means inserting object into index is enough to persist the object. The colorally is: removing the index entries referring to an object from all indexes will also de-allocate the obbject during the next garbage collection cycle (assuming it's not reachable from other reachable persistent objects).

Most index creation functions are generic and require providing type of the value, which must implement IPersistent interface. Most of the time you'll create homogenous index (i.e. indexes where all values are of the same type). You can create heterogenous indexes (i.e. allowing values of different types) by using IPersistent as the type of the value.

Types of indexes

Volante offers several kinds of indexes so it's important to choose the right index for your needs.

B-Tree indexe is the most widely used. It keeps the objects sorted by a key. A key can be of any basic type (e.g. short, int, string, DateTime). That allows quickly finding an object having a given key or find all objects in a given range. Indexes can be unique (do not allow duplicate keys) or non-unique (allow duplicate keys). A thick index offers a more efficient implementation if there are many duplicate keys.

In a generic B-Tree index the key can be of any value, not necessarily related to the object stored under that key. Field and multi-field indexes are different in that the key value must be part of the object.

Spatial index is for geometric data. The key is a rectangle (either with integer or floating-point coordinates).

Sorted collections is similar to generic B-Tree index but it allows (and requires) providing a custom comparator which determines the order of keys.

Set is used to store a set of objects (i.e. a collection of objects without duplicates).

Link provides one-to-many mapping.

Array is an array of persistent objects. Like in regular array, objects are reffered to by an index within array.

Relation is used to represent a relation between objects. It allows performing what in relational terms is called a join.

Time series is used to efficientl store time series data.

Patricia tree offers memory-efficient storage of keys by compressing common paths of the key.
